What is whiteboarding?
Whiteboarding can be defined as an application data sharing protocol for graphics applications. It is composed of four separate activities/formats
  1. User handling. User (two or more) joining and leaving a whiteboarding session
  2. Sending messages with data to each other
  3. Server keeping the current state of the whiteboard
  4. Data formats
The three first activities could be regarded as the protocol carrying any data (4). Therefore, the four things above can be simplified to:
  1. Protocol
  2. Data formats
By proposing a good protocol the Whiteboarding JIG proposal can later be used for any application data. Why don't make a collaborative spreadsheet? Or a 3D modeling tool. Or a real-time strategy game?

Conferencing and user-to-user
Whiteboarding should be supported both during user-to-user chats and conferencing. The latter adds some complexity since users should be able to join an ongoing whiteboarding session. This means that the server needs to have a history that keeps the current state of the whiteboard and sends that to users entering the room.

Two initial data formats
In the first step, two different formats are needed: one for bitmap images and one for vector graphics. For bitmaps there is already a proposed format for Jabber, a compressed derivative of XPM (http://docs.jabber.org/draft-proto/html/sxpm.html). For vector images I propose to use SVG (http://www.w3.org/TR/SVG/). SVG is currently a W3C Recommendation for vector graphics and is therefore very likely to get a very wide support. SVG also include standards for, for example animations, metadata and scripting.

Building a renderer and editor for SVG is not a simple task for Jabber client developers. But, there is already a set of available libraries that can help. For exemple, Windows and web developers can use the Adobe SVG Viewer ActiveX (http://www.adobe.com/svg) and for Java there is the Batik SVG toolkit (http://xml.apache.org/batik).

The problem with SVG is it's complexity, it has a much bigger scope then needed for whiteboarding, therefore the whiteboarding-JIG should decide on a subset of SVG to use. Also, SVG will need a few additions, for example a z-index attribute and ownership/access rights for each element. These things are easy to add since SVG allows for keeping foreign data using separate namespaces.

Transports
There are currently several tools for whiteboarding, for example Microsoft Netmeeting (integrated with MSN Messenger, http://www.microsoft.com/windows/NetMeeting/Features/Whiteboard) and Groove (http://www.groove.net/). There is also a standard for whiteboarding for multimedia telecom conferencing purposes, T.126 (http://www.imtc.org/t120.htm). The Jabber whiteboard should be able to communicate with these whiteboarding solutions via Jabber server transports. The protocol should also allow for other data formats (for example spreadsheets) to communicate with external application using other transports.

Adding Jabber to existing applications
I think there are several existing applications that would greatly benefit from getting support for collaboration. What about being able to share, in real-time, a drawing in Photoshop with a friend who's using Gimp? This should be possible to do via Jabber, now wouldn't that be cool? :-)
